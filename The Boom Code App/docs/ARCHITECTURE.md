# ARCHITECTURE — The Boom Code App

## Overview

A React single-page application deployed as a **Power Apps Code App** into the og-dv Dataverse environment. Provides full CRUD for 8 Dataverse tables through a config-driven, generic component architecture.

## Layer Architecture

### 1. Pages Layer (`src/pages/`)

Thin wrappers that import an `EntityConfig` and render `<EntityListPage>`. Each page is ~3 lines:

```tsx
import { EntityListPage } from "@/components/entity-list-page";
import { ACCOUNT_CONFIG } from "@/types/entities";
export default function AccountsPage() {
  return <EntityListPage config={ACCOUNT_CONFIG} />;
}
```

### 2. Generic Component Layer (`src/components/`)

- **`EntityListPage`** — Config-driven CRUD page with search, data table, inline actions
- **`RecordFormDialog`** — Dynamic form dialog that renders fields based on `FieldConfig` type (text, textarea, number, date, datetime, optionset, lookup)
- **`ConfirmDeleteDialog`** — Simple delete confirmation
- **`SidebarNav`** — Navigation driven by `ALL_ENTITY_CONFIGS`

### 3. Hooks Layer (`src/hooks/use-dataverse.ts`)

React Query hooks that wrap the data access layer:
- `useEntityList<T>(config, options)` — fetch list with optional filter/top
- `useEntityRecord<T>(config, id)` — fetch single record
- `useCreateEntity<T>(config)` — mutation for create
- `useUpdateEntity(config)` — mutation for update
- `useDeleteEntity(config)` — mutation for delete
- `useLookupOptions(field)` — fetch options for lookup dropdowns

All mutations auto-invalidate queries and show toast notifications.

### 4. Data Access Layer (`src/lib/dataverse.ts`)

A `serviceMap` registry maps `entitySetName` → generated service class. Exposes five functions:
- `listRecords<T>(config, options)`
- `getRecord<T>(config, id)`
- `createRecord<T>(config, data)`
- `updateRecord(config, id, data)`
- `deleteRecord(config, id)`
- `fetchLookupOptions(entitySetName, keyField, displayField)`

### 5. Generated SDK Layer (`src/generated/`)

Auto-generated by `pac code add-data-source -a dataverse -t <logicalName>`. Each table gets:
- `services/<Table>Service.ts` — typed CRUD class using `getClient(dataSourcesInfo)`
- `models/<Table>Model.ts` — TypeScript interface for the table shape

### 6. Configuration Layer (`src/types/entities.ts`)

Central config file defining:
- **Entity interfaces** — TypeScript types for each table
- **Option set constants** — value/label arrays
- **`EntityConfig` objects** — entitySetName, primaryKey, fields[], routes, icons
- **`FieldConfig`** — field-level metadata: type, lookup bindings, option sets, display flags

## Key Design Decisions

| Decision | Rationale |
|----------|-----------|
| Config-driven generic components | One `EntityListPage` serves all 8 tables — zero per-entity component code |
| Database references, not connectors | Connector pattern silently fails; database references use correct SDK path |
| No `$select` | Computed fields cause silent failures; omitting is safer at our scale |
| React Query | Automatic caching, refetch, optimistic updates, loading/error states |
| OData `contains()` filter for search | Server-side filtering on primary name field |
| Lookup bind/unbind via `@odata.bind` | Standard Dataverse OData write pattern for navigation properties |

## Build & Deploy

```bash
npm run build    # tsc -b && vite build → dist/
pac code push    # uploads dist/ to Power Apps
```

Bundle size: ~700KB (single chunk). Could be split with lazy routes if needed.
